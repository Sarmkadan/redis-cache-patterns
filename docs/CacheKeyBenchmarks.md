# CacheKeyBenchmarks

The `CacheKeyBenchmarks` class serves as a centralized container for predefined cache key strings used throughout the `redis-cache-patterns` project, specifically designed to support performance benchmarking scenarios. It exposes a set of public string fields representing distinct caching patterns—such as user-specific data, product details, inventory states, and collection queries—allowing benchmark suites to consistently reference and measure the latency and throughput of various Redis operations without hardcoding literal strings across test methods.

## API

### UserKey
*   **Type**: `public string`
*   **Purpose**: Holds the standardized cache key pattern used for storing and retrieving user-specific data during benchmarks.
*   **Parameters**: None (field).
*   **Return Value**: Returns the string value assigned to the user key pattern.
*   **Throws**: None. Accessing this field does not throw exceptions unless the instance itself is null.

### ProductKey
*   **Type**: `public string`
*   **Purpose**: Holds the standardized cache key pattern used for storing and retrieving product catalog information during benchmarks.
*   **Parameters**: None (field).
*   **Return Value**: Returns the string value assigned to the product key pattern.
*   **Throws**: None.

### InventoryKey
*   **Type**: `public string`
*   **Purpose**: Holds the standardized cache key pattern used for storing and retrieving real-time inventory counts or status during benchmarks.
*   **Parameters**: None (field).
*   **Return Value**: Returns the string value assigned to the inventory key pattern.
*   **Throws**: None.

### GenericBuildKey
*   **Type**: `public string`
*   **Purpose**: Holds a generic cache key pattern intended for benchmarking build artifacts or computed results that do not fit specific domain entities.
*   **Parameters**: None (field).
*   **Return Value**: Returns the string value assigned to the generic build key pattern.
*   **Throws**: None.

### EntityKey
*   **Type**: `public string`
*   **Purpose**: Holds the standardized cache key pattern used for generic entity retrieval operations, typically testing polymorphic or base-class caching strategies.
*   **Parameters**: None (field).
*   **Return Value**: Returns the string value assigned to the entity key pattern.
*   **Throws**: None.

### PatternKey
*   **Type**: `public string`
*   **Purpose**: Holds a cache key string specifically designed to test Redis pattern matching capabilities (e.g., keys used with `KEYS` or `SCAN` commands) during performance analysis.
*   **Parameters**: None (field).
*   **Return Value**: Returns the string value assigned to the pattern test key.
*   **Throws**: None.

### CollectionKey
*   **Type**: `public string`
*   **Purpose**: Holds the standardized cache key pattern used for storing and retrieving collections or lists of items during benchmarks.
*   **Parameters**: None (field).
*   **Return Value**: Returns the string value assigned to the collection key pattern.
*   **Throws**: None.

## Usage

The following examples demonstrate how to utilize the `CacheKeyBenchmarks` class within a benchmarking context to ensure consistent key usage across different test scenarios.

### Example 1: Initializing Benchmark Scenarios
This example shows how to instantiate the class and retrieve specific keys to configure a benchmark runner for user and product data.

```csharp
using RedisCachePatterns.Benchmarks;

public class CacheBenchmarkSetup
{
    public void ConfigureScenarios()
    {
        var keys = new CacheKeyBenchmarks();
        
        // Configure the benchmark runner with specific key patterns
        var userScenario = new BenchmarkScenario
        {
            Name = "UserLookupLatency",
            CacheKey = keys.UserKey,
            OperationType = CacheOperation.Get
        };

        var productScenario = new BenchmarkScenario
        {
            Name = "ProductCatalogThroughput",
            CacheKey = keys.ProductKey,
            OperationType = CacheOperation.Set
        };

        // Execute scenarios...
    }
}
```

### Example 2: Validating Key Consistency in Tests
This example illustrates using the fields to assert that the correct key patterns are being generated by a key factory during a test run.

```csharp
using RedisCachePatterns.Benchmarks;
using Xunit;

public class KeyFactoryTests
{
    [Fact]
    public void InventoryKey_MatchesExpectedPattern()
    {
        var benchmarks = new CacheKeyBenchmarks();
        var factory = new CacheKeyFactory();

        // Assume the factory generates keys based on a known enum or type
        var generatedKey = factory.CreateKey(CacheEntityType.Inventory);

        Assert.Equal(benchmarks.InventoryKey, generatedKey);
    }

    [Fact]
    public void CollectionKey_UsedForListOperations()
    {
        var benchmarks = new CacheKeyBenchmarks();
        
        // Simulate a bulk operation test
        var testKeys = new List<string> 
        { 
            benchmarks.CollectionKey, 
            benchmarks.EntityKey 
        };

        Assert.Contains(benchmarks.CollectionKey, testKeys);
    }
}
```

## Notes

*   **Immutability and Thread Safety**: As `CacheKeyBenchmarks` exposes its data via public mutable fields (`public string`), the instance itself is not thread-safe if these fields are modified after initialization. In a multi-threaded benchmarking environment, it is recommended to treat instances as immutable once constructed or to ensure that each thread operates on its own instance of `CacheKeyBenchmarks` to avoid race conditions where a key value might change mid-execution.
*   **Null Handling**: The class definition does not enforce non-null values for these strings via the signature alone. Consumers should verify that the fields are initialized (non-null) before passing them to Redis client methods to prevent `ArgumentNullException` in downstream operations.
*   **Key Pattern Integrity**: The `PatternKey` field is intended for use with Redis pattern-matching commands. Care should be taken when modifying this value in derived classes or external configurations, as invalid Redis patterns (e.g., missing wildcards where expected by the test logic) may result in empty result sets or performance degradation during `SCAN` operations.
*   **No Parameter Validation**: Since these are direct field exposures, no validation logic is executed upon assignment. It is the responsibility of the code assigning values to these fields to ensure they conform to Redis key naming constraints (e.g., avoiding excessive length or forbidden characters) to prevent runtime errors during actual cache operations.
